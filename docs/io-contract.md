# Input Output Contract

This is the exact folder contract expected by UNav mapping scripts. The names must match the command arguments exactly.

## Parameter Definitions

Command format:

```bash
python -m unav.run_mapper <data_temp_root> <data_final_root> <feature_model> <place> <building> <floor>
```

- `data_temp_root`: root folder for raw mapping inputs and intermediate processing artifacts. This is where the raw floor video goes before processing.
- `data_final_root`: root folder for final map artifacts used by localization/navigation. This is where the floorplan, features, COLMAP map, alignment matrix, and labels live.
- `feature_model`: global descriptor model, usually `DinoV2Salad` unless the deployment uses another supported model.
- `place`: city/campus/region namespace, for example `New_York_City`.
- `building`: building namespace, for example `LightHouse`.
- `floor`: floor namespace and video basename, for example `4_floor`.

Recommended naming rules:

- Use ASCII letters, numbers, underscores, and hyphens.
- Avoid spaces in `place`, `building`, and `floor`.
- The video filename must match the `floor` argument: `<floor>.mp4`.

## Required Raw Mapping Inputs

### Under `data_temp_root`

```text
<data_temp_root>/
├── orb_vocab.fbow
└── <place>/<building>/<floor>/
    └── <floor>.mp4
```

Example:

```text
/mnt/data/UNav-IO/temp/
├── orb_vocab.fbow
└── New_York_City/LightHouse/4_floor/
    └── 4_floor.mp4
```

`<floor>.mp4` should be the exported spherical/equirectangular 360 video from the Insta360 capture workflow. Do not put `.insv`, `.360`, `.lrv`, `.thm`, or reframed flat videos here as the mapping input.

## Required Floorplan Inputs

### Under `data_final_root`

```text
<data_final_root>/
├── scale.json
└── <place>/<building>/<floor>/
    └── floorplan.png
```

Example:

```text
/mnt/data/UNav-IO/data/
├── scale.json
└── New_York_City/LightHouse/4_floor/
    └── floorplan.png
```

The floorplan must correspond to the same physical floor as the video. `scale.json` stores meters-per-pixel values used by the aligner.

## Generated Intermediate Outputs

The mapping pipeline writes intermediate files under `data_temp_root`:

```text
<data_temp_root>/<place>/<building>/<floor>/
├── stella_vslam_dense/
│   ├── final_map.msg
│   ├── output_cloud.ply
│   ├── keyframes/
│   └── eval_logs/keyframe_trajectory.txt
├── perspectives/*.png
└── colmap_sfm/
    ├── sparse/0/cameras.txt
    ├── sparse/0/images.txt
    ├── pairs.txt
    ├── matches.h5
    └── database.db
```

These are useful for debugging and re-running stages, but they are not the final customer deliverable by themselves.

## Generated Final Outputs

The final map package for each floor lives under `data_final_root`:

```text
<data_final_root>/<place>/<building>/<floor>/
├── colmap_map/                  # triangulated map output
├── features/
│   ├── local_features.h5
│   └── global_features_<model>.h5
├── floorplan.png
├── transform_matrix.npy         # generated after aligner step
└── boundaries.json              # generated by labelme map labeling step
```

For multilingual labels, the translation GUI also writes:

```text
<data_final_root>/_i18n/labels.json
```

## Minimal Deliverable for Navigation

For each floor, the final handoff should include:

- `floorplan.png`
- `boundaries.json`
- `transform_matrix.npy`
- `features/local_features.h5`
- `features/global_features_<model>.h5`
- `colmap_map/`

For multilingual deployments, also include:

- `_i18n/labels.json`

## Upload or Handoff Checklist

Before uploading to UNav Cloud or handing the files to the backend team, verify:

- The `place/building/floor` folder names match the intended deployment names.
- `boundaries.json` loads in `labelme` without errors.
- `transform_matrix.npy` was generated from the matching floor video and floorplan.
- Feature files are present and non-empty.
- Destination labels in `boundaries.json` match the customer-facing destination list.
- `_i18n/labels.json` is present if multiple languages are required.
